# Doodiary — 設計判断の記録

このアプリを作るにあたって検討した技術選定・事業判断とその理由をまとめる。
「なぜこの選択をしたか」を後から見返すためのドキュメント。

---

## 1. AI（分析エンジン）

### 結論
**Claude API（Haiku 4.5）を使う。**

### 検討した選択肢

| アプローチ | 評価 |
|---|---|
| LLM（Claude） | ◎ 採用 |
| LLM（Gemini Flash） | △ コストは安いが日本語の繊細さで劣る |
| 専用感情分析API（Google NL / AWS Comprehend） | ✗ テキスト前提で使えない |
| オンデバイスML（TFLite） | △ 即時スコア計算には使う、深い分析は不可 |

### なぜClaudeか
- **入力がテキストではない。** スライダー値・色・天気enum・座標・画像という異種データを統合解釈できるのはLLMだけ。専用感情分析APIはテキスト前提なので不適。
- **メンタルヘルス文脈の繊細さ。** 「今日のあなたへのメッセージ」のような共感的で自然な日本語生成はClaudeが優秀。Geminiより安全性設計が丁寧。
- **心理フレームワークの埋め込み。** CBT/ACTをsystem promptに入れたときの忠実度がClaudeは高い。

### なぜHaikuか
- 1リクエストのコストが約$0.002（input ~500 / output ~300 tokens）。
- 精度が足りなければSonnetやOpusに上げられる（Pro機能の深い分析だけ上位モデル、という出し分けも可能）。

### コスト試算（Haiku, 1ユーザー1日1回）
- 約¥9/月/ユーザー
- 10,000 DAU でも月¥90,000程度

## 2. バックエンド

### 結論
**Supabase を使う。**

### Firebase との比較

| 項目 | Firebase | Supabase |
|---|---|---|
| DB | NoSQL（Firestore） | PostgreSQL（SQL）◎ |
| 料金 | 従量課金、読みにくい | 予測しやすい ◎ |
| Edge Functions | Cloud Functions（コールドスタート遅い） | Deno、速い ◎ |
| ロックイン | Google依存 | OSS、自前ホスト可 ◎ |
| リアルタイム | 優秀 ◎ | Realtimeで対応可 |

### なぜSupabaseか
- 感情記録は**リレーショナルに集計したい**（「直近7日の平均energy」のようなクエリがSQLで自然に書ける）。Firestoreだと集計が面倒。
- 料金が予測しやすく、MVPは無料枠で賄える。
- 将来のグループチャットもSupabase Realtimeで対応できる。

## 3. 認証

### 結論
**Supabase Auth に一本化。匿名ログイン → 後からアカウント昇格。**

### なぜこの設計か
- メンタルヘルス系アプリは、最初から個人情報を求めると離脱する。
- 初回は匿名ログインで即使える状態にする。
- 7日経過 or Pro誘導のタイミングで「データを引き継ぐには登録が必要」とApple/Google昇格を促す。**登録を強制しない=離脱を防ぐ**、かつ**データ引き継ぎを動機に登録させる**自然な導線。

## 4. 課金

### 結論
**RevenueCat（アプリ内課金）。**

### なぜか
- App Store / Google Play の規約上、デジタルサブスクはIAP必須。
- RevenueCatでiOS/Androidの課金を一元管理。無料トライアル設定も簡単。10,000アクティブユーザーまで無料枠。
- 手数料は各ストア30%（初年度15%減額制度あり）。¥600/月なら実質¥420が手元に。
- 課金状態はSupabaseにも保存して整合を取る。

## 0. このアプリの核となる思想（最重要）

**「ユーザーに卒業してもらうこと」がゴール。**

```
悩まないことが大事
→ 自分に向き合えるようになる
→ もうアプリが要らなくなる
→ 解約してもらう。それでいい。
```

良いカウンセラーと同じ思想。良いセラピストはクライアントが自分を頼らなくなることを目指す。
**依存させない。卒業を歓迎する。** これがDoodiaryのすべての判断の土台になる。

### この思想が各設計に与える影響

| 領域 | 一般的なアプリ | Doodiaryの正解 |
|---|---|---|
| 継続 | 年額で縛る（lock-in） | 縛らない。月額中心 |
| 解約 | 全力で防ぐ | しやすくする・責めない |
| KPI | 滞在時間・継続率を最大化 | ユーザーが楽になることが成功 |
| 通知 | 毎日開かせる | 「最近調子いいのかも」とそっと卒業を後押し |

- **年額で安く釣って縛るのは、この思想と矛盾する。** 年払いして3ヶ月で卒業したら、使わない9ヶ月分を取ることになる。それは「人の助けになりたい」と逆。
- 解約時に責めない（「また辛くなったら戻ってきてね」）。
- 機能を作るとき・コピーを書くとき・通知を設計するとき、常に「これはユーザーの自立を助けるか、依存させるか」で判断する。

## 5. マネタイズ設計

### 価値観との整合
当初「過去データを全部有料化」を検討したが、Yusukeの価値観（**人の助けになりたい**）と衝突した。
→ **無料でも当日・過去7日は全部使える**形に修正。続けて価値を感じた人が払う、強制しない設計に。

### 料金の方針（卒業思想を反映）
- **月額 ¥600 をメインにする。** いつでも辞められる、解約に優しい設計。
- **年額で縛らない。** 置くとしても軽い割引のみ（お得さで釣らない）。続けたい人だけが選ぶ穏やかな選択肢。
- 解約フローで責めない。卒業をそっと後押しする通知も検討。

### プラン
```
無料
├── 記録：無制限
├── 当日のグラフ・インサイト
└── 過去7日分

Pro（¥600/月 or ¥4,800/年）
├── 過去データ無制限
├── 週次・月次インサイト
├── パターン検出
└── 将来：グループ機能
```

年額 ¥4,800 = 月換算 ¥400（月額の約2/3、4ヶ月分無料相当）。継続を強く促す価格にした。

### 収益試算（転換率3%・月額¥600想定）

| DAU | 月売り上げ | APIコスト | 粗利 |
|---|---|---|---|
| 1,000 | ¥18,000 | ¥9,000 | ¥9,000 |
| 5,000 | ¥90,000 | ¥45,000 | ¥45,000 |
| 10,000 | ¥180,000 | ¥90,000 | ¥90,000 |

※ Proユーザーだけにインサイト生成を絞ればAPIコストは大幅に下がり、粗利率はさらに上がる。

※ Proユーザーだけにインサイト生成すればAPIコストはさらに下がる。

### 最重要
**課金転換率はアプリの「続けたい」度で決まる。** まず毎日開きたくなるUXを作ることが最優先。
